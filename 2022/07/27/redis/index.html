1、NoSQL

Not Only SQL 泛指非关系型数据库，不依赖于业务逻辑的方式存储，以键值对的形式存储。

  1. 不遵循SQL标准
  2. 不支持ACID
  3. 性能远超SQL

使用场景

  1. 高并发
  2. 海量数据读写
  3. 数据高扩展性
  4. 用不着SQL或用了SQL也不行的场景

不适应场景

  1. 事务支持
  2. 基于SQL的结构化存储

Redis特性

  1. 支持多种数据类型
  2. 对于数据类型的操作多样，且具有原子性
  3. 支持各种不同方式的类型
  4. 数据缓存在内存中，周期性将数据写入磁盘或记录修改日志
  5. 实现主从同步

2、常用数据类型

2、String 字符串

二进制安全，可以保存二进制文件，value最多可以为512Mb

value底层采用动态字符串，字符串大小小于1m，扩容时翻倍，大于1m时，扩容时增加1m，最大512m

redis操作是原子性的

3、List 列表

单键多值，按插入顺序排序，可以添加数据到头尾，底层实现为双向链表，当元素较少时使用连续分配空间的压缩列表，当储存大量数据时使用多个压缩列表组成大列表

4、Set 集合

类似于list，但可以自动排重，且无序，底层实现为value为空的hash表所实现的字典，添加删除查找的时间复杂度为O(1)

5、Hash 哈希

值是一个string类型f的filed和value的映射表即键值对的集合，适合存储对象，类似java的Map<String, Object> ，底层实现为两种，当field较少时使用压缩列表，当field较多时使用哈希表

key {field1=value1，field2=value2}

6、Zset 有序集合

类似于set，但是每个成员关联一个“评分”，被用于按照从最低分到最高分排序，成员唯一，但评分不一定唯一。底层实现类似于Map<String, Double>，又类似于TreeSet，根据权重排序，使用两种结构，hash关联value和score，跳跃表用于给value排序

8、事务和锁机制

单独隔离的操作，事务中的命令都被序列化、按序执行，执行过程中不会被其他命令打断

组队时有命令错误导致失败，命令执行都失败

组队时有错误命令但成功组队，只有错误的命令无法执行

悲观锁：每次操作前上锁，操作结束后释放锁

乐观锁：每次操作不会上锁，但是在执行更新的时候会判断一下操作期间是否有人修改，有人修改则此次操作失败。可以使用版本号等机制。redis采用乐观锁

Redis事务三重性

1、单独的隔离操作

2、没有隔离级别

3、不保证原子性，一条命令执行失败，其他命令仍被执行，没有回滚

秒杀案例

秒杀过程：库存减一，添加入用户Id

使用乐观锁，淘汰用户

关于乐观锁造成的库存遗留问题：高并发的情况下，只有一人能修改数据，导致其他用户即使有库存的情况下也无法秒杀。使用Lua等脚本语言，在库存有剩余的情况下再次秒杀

9、持久化操作

1、RDB

redis数据备份文件

在指定时间间隔内将内存中的数据写入磁盘，使用Snapshot机制，默认保存在运行目录

主动关闭redis时会执行一次持久化，开启redis时会读取持久化文件

RDB机制

RDB的fork原理

bgsave会fork主进程得到子进程，共享主进程内存数据，此时主进程阻塞直到fork完成，fork完成后读取内存数据并写入RDB文件

copy-on-write技术

  1. 当主进程执行读操作时，访问共享内存
  2. 当主进程执行写操作时，则会拷贝修改的数据，在备份上进行写操作
  3. 子进程进行复制操作时，共享内存是只读状态

2、AOF

追加文件

redis的每一个写命令都会记录在AOF文件中，可以看作命令日志文件，由主进程执行

AOF默认关闭，需要修改配置文件

AOF记录频率

解决文件过大和同一key的多次操作

通过执行bgrewriteaof命令，让AOF执行重写，用最少的命令达到效果

redis在触发阈值时，也会自动执行重写，可以在配置文件中设置

3、混合持久化

将rdb文件和增量aof放在一起，aof不再记录所有的命令，而是记录两次RDB持久化之间的命令，重启redis时优先加载rdb文件，在加载aof文件

10、缓存设计

1、缓存穿透

现象：

  1. 应用服务器压力增大
  2. redis命中率降低，即缓存中查不到数据
  3. 一直查询数据库，数据库无法同步缓存

原因：

  1. redis查询不到数据，自身业务逻辑存在问题
  2. 大量非正常url访问

解决方法：

  1. 缓存空值：如果查询缓存数据为空，则将该空值进行缓存，并将过期时间设置为一个较短时间
  2. 设置白名单：使用bitmap定义一个可访问白名单，名单id作为bitmaps的偏移量，每一次访问和map中的id进行比较，id存在才可访问
  3. 布隆过滤器：连续的数据结构，每个储存位都是一个bit，用以标识数据是否存在。存储数据时，使用多个hash函数将这些变量映射为多个点。判断key是否存在时，使用这些hash函数计算，判断对应的储存位是否为1
  4. 实时监控：Redis命中率急剧降低时，排查访问对象，进行处理

2、缓存击穿

现象：

  1. 数据库压力瞬间增加
  2. reids没有大量key过期
  3. redis正常运行

原因：

  1. 一个并发量较大的key在某个时间过期，导致所有请求发送到数据库中

解决方法：

  1. 预先设置热门key
  2. 实时调整，实时调整redis的key过期时间
  3. 使用锁：查询某个key如果为空，则对这个key设置排他锁，同时取去数据库查询，写入缓存后，在释放锁。在上锁期间，请求将被阻塞，释放锁后，后面的所有请求都可以从缓存中拿到数据

3、缓存雪崩

现象：

  1. 数据库压力变大
  2. 服务器崩溃

原因：

  1. 在极少时间段出现大量key过期

解决方案：

  1. 构建多级缓存，比如Nginx缓存、redis缓存
  2. 使用锁或者队列，不适合高并发
  3. 及时更新缓存，某个key将要过期时，触发通知另一个线程去后台实时更新key
  4. 将缓存失效时间分散开，在原有缓存失效时间上增加一个随机值，这样每一个缓存的过期时间的重复率将降低，很难引发集体失效

4、数据库与缓存的一致性

缓存不一致的原因：

  1. 缓存key删除失败
  2. 并发写入脏数据

选择合适的更新缓存的策略

  1. 以删除缓存代替对缓存的写操作
  2. 先更新数据库，再删除缓存

缓存不一致的处理：

  1. 设置key过期时间作为最基本的保证
  2. 延时双删：第一次删除缓存后，更新数据库，过一段时间再次删除
  3. 监听数据库变化，获取需要操作的数据，将缓存放入消息队列，用消息队列的重试机制，重试删除对应的key

11、Redis集群

进一步提高redis并发能力，需要搭建主从集群，实现读写分离，主节点写从节点读

启动多个redis实例后，开启主从关系：

  1. 修改配置文件（永久配置）：添加命令slaveof <masterip> <masterport>
  2. 连接redis服务后，客户端执行命令slaveof <masterip> <masterport>

数据同步原理：

两个概念：repid，数据集标志，id一致则是统一数据集，每个主节点有唯一id，从节点继承主节点。offset，偏移量，记录在repl_baklog文件中的命令会随着数据量增大而增大，从节点完成同步时会记录当前偏移量，如果从节点偏移量小于主节点，则从节点数据落后需要更新

主从第一次同步执行全量同步：从节点发送请求增量同步，主节点判断replid，发现不一致，拒接增量同步并发送版本信息，从节点保存主节点数据的版本信息，主节点执行bgsave生成RDB，并将RDB文件发送给从节点，从节点加载RDB。在主节点bgsave阶段的写命令会被保存到repl_baklog文件中，RDB完成后该文间发送给从节点，从节点执行命令完成同步。

主从增量同步：从节点发送请求增量同步并发送repid和偏移量，主节点判断replid，并根据偏移量发送对应的repl_baklog文件数据，从节点执行命令。增量同步失败：repl_baklog文件超过上限，最开始的命令被覆盖，从节点无法正常同步则执行全量同步

优化数据同步：

  1. 无磁盘复制，避免磁盘IO：配置文件中修改repi-diskless-sync yes
  2. 控制单节点内存大小，减少RDB导致的磁盘IO
  3. 适当提高repl_baklog的大小，从节点宕机尽快回复，避免全量同步
  4. 限制主节点上的从节点，从节点过多可以使用主-从-从的链式结构

1、哨兵模式

Redis提供哨兵（Sentinel），实现主从集群的的自动故障回复

哨兵作用

  1. 监控：Sentinel会不断检查主从节点是否按预期工作
  2. 自动恢复：如果主节点故障，sentinel会将一个从节点升级为主节点，当故障节点恢复后，以新的主节点为主

服务状态监控：

sentinel基于心跳机制检测服务，每隔一秒向每个实例发送ping命令

  1. 主观下线：如果sentinel节点发现某个实例未在规定时间响应，则认为该实例主观下线
  2. 客观下线：如果超过指定数量（quorum）的sentinel都认为该实例主观下线，则该实例客观下线。该数量最好设置为sentinel实例数量的一半

选择新的master：

  1. 先过滤主观下线、断线、5秒内没有回应sentinel节点ping响应、与主节点失联超过指定值（down-after-miliseconds * 10）的从节点
  2. 然后判断从节点的slave-property，越小优先级越高，为0永不选举
  3. 然后判断从节点的offset，越大则数据越新，优先级越高
  4. 最后判断从节点运行id，越小优先级越高

故障转移：

  1. sentinel向选举的从节点发送命令，slaveof no one，成为主节点
  2. sentinel向其他节点广播slaveof <master> <masterport>命令，从节点同步新主节点的数据
  3. sentinel修改故障节点，让其成为新主节点的从节点

2、分片集群

实现数据分区：突破单机限制，容量提高；每个节点均可以对外提供读写服务，提高集群响应能力

实现高可用：支持自动故障转移，任意节点故障，集群仍可对外提供服务

分片集群特征：

  1. 多个主节点，每主节点r保存不同数据
  2. 每个主节点有多个从节点
  3. 主节点之间通过ping检测健康状态，节点下线流程同哨兵模式
  4. 客户端可以访问任意节点，最终被转发到正确节点

散列插槽：

Redis会将每一个master节点映射到0~16383插槽上，节点存储的数据不与节点绑定而与插槽绑定。

redis根据key的有效部分（例如：key是num，有效部分为num；key是{item}num，有效部分就是item）计算插槽值，根据插槽值寻找对应的节点存储查找数据。

将同一类数据保存到固定节点时，使用相同的有效值

故障转移：

redis分片集群，主机宕机后自动故障转移

部署redis集群应至少部署在三个物理节点上，才能避免单点问题

12、Redis应用

1、分布式锁

单机部署系统演化为分布式系统，分布式系统多线程、多进程且分布在不同的机器上，原本单机部署的控制锁策略将失效，则需要分布式锁解决

实现方案：redis实现分布式锁

多个客户端使用setnx指令，获取到key后，其他客户端无法再设置key，相当于上了锁，完成业务逻辑后删除key，相当于释放锁。为防止死锁，需要设置过期时间

可以使用Redission框架实现分布式锁

2、异步队列

  1. list作为队列，lpush生产消息，brpop消费信息：当list为空时brpop会被阻塞，直到list中有值或超时。这种方式只能实现一对一的消息队列。

  2. 使用pub/sub进行消息的发布/订阅

一般异步队列的实现还是交由专业的消息队列

3、延时队列

使用zet，将时间戳作为score进行排序，可以一直向内存中生产消息，再利用zrangebyscore查询符合条件的待处理任务，再循环执行队列任务即可

4、lua脚本

Redis的事务功能比较简单，平时的开发中，可以利用Lua脚本来增强Redis的命令。

Lua脚本能给开发人员带来这些好处：

  • Lua脚本在Redis中是原子执行的，执行过程中间不会插入其他命令。
  • Lua脚本可以帮助开发和运维人员创造出自己定制的命令，并可以将这 些命令常驻在Redis内存中，实现复用的效果。
  • Lua脚本可以将多条命令一次性打包，有效地减少网络开销。

5、redis管道

Redis 服务端接收到管道发送过来的多条命令后，会一直执命令，并将命令的执行结果进行缓存，直到最后一条命令执行完成，再所有命令的执行结果一次性返回给客户端。

优点：

  1. 减少服务器与客户端的IO
  2. 减少系统调用的次数
